# Wager Mode — devnet-SOL stakes (feature branch design)

**Branch:** `worktree-wager-devnet-sol` — NEVER merged into the hackathon submission.
**Status:** designed via audited two-design judge panel (2026-07-04); implementation in flight.

## Goal

Optional per-group **wager mode**: members back/doubt claims with real devnet SOL —
real on-chain deposits, escrowed stakes, on-chain withdrawals — exercising full
real-money mechanics with a worthless token. Rep mode stays default and byte-identical.

## Hard constraints

- Custodial v1: engine-held wager treasury (SEPARATE keypair from the TxLINE/TxL
  wallet — sponsor terms forbid TxL as wagering collateral). No custom on-chain program.
- Feature-flagged: `WAGER_MODE_ENABLED` env master switch + per-group admin opt-in.
- Escrow-then-internal-stake (Telegram can't sign; per-stake on-chain payment would
  break the 150s anti-snipe pending window).
- Deny-list-compliant copy everywhere EXCEPT strings scoped inside `wager/copy.ts`.
- All lamports as safe integers; idempotent deposits (keyed `(tx_sig, ix_index)`) and
  idempotent sends (deterministic pre-broadcast signature + `searchTransactionHistory:true`).
- Solvency: per-market liability ledger; stakes refused beyond covered treasury exposure;
  persisted circuit breaker.
- Public web surface must never leak pubkeys or per-user lamports.
- **Migration 0002 is NOT applied to the live demo Supabase project** — spin a separate
  project (or apply after the hackathon).

## Judge verdict (summary)

Design B wins the two axes that matter most — money safety and hackathon contamination — but ships four concrete mechanical bugs that Design A got right, so the answer is a synthesis on B's skeleton.

ATTACKS ON A (minimal delta, shared ledger): (1) CONFIRMED against code: balance() at packages/db/src/engine-db.ts:271-281 sums ALL ledger_entries for (group,user) with no filter. A's shared-table-plus-currency-column plan forces an edit to this hot Rep query and makes every future ledger read a foot-gun: one missed currency filter and a 10,000,000-lamport deposit prints as 10M Rep on the flagship leaderboard. Silent-corruption class, full blast radius on the hackathon economy. (2) A mutates the Rep path in at least six places (debit-first ordering for BOTH currencies, client-generated position uuids, orphan-stake sweeper over Rep rows, extended settlement sweeper re-posting Rep ledger entries, settle() void-refund change, ledger_entries kind CHECK drop/re-add) — violating the fixed constraint "Rep mode untouched and default" and destroying the delete-the-module-get-main property; rebases against main become minefields. (3) Its stake serialization is an in-process withKeyLock: Railway rolling deploys briefly run two instances, so a double-tap across the overlap overdraws SOL escrow — a real double-spend, exactly the race the brief names. (4) Its insolvency circuit breaker is in-memory: a crash-loop re-arms stake acceptance while insolvent until the next 5-min check. (5) wallet_links.deposit_group_id ("deposits land in whichever group you last ran /deposit in") is a support dead-end: deposit sent hours after linking lands in the wrong group's pot with no transfer command. ATTACKS ON B (clean separation): (1) wager_deposits PK = tx_sig alone drops lamports when one transaction carries multiple system transfers to the treasury (CLI/dapp batch) — A's (tx_sig, ix_index) key is correct. (2) The insertMarket→stampMarketIfWagerGroup two-step has a crash window minting a Rep market in a SOL group, and isWagerMarket() costs an extra roundtrip on every stake tap and settle — verified that both handleStake (callbacks.ts:464) and settle() (settler.ts:191) already hold the full MarketRow, so A's markets.currency column (default 'rep', stamped atomically in the same insert) is strictly better: no window, zero extra queries, invisible to Rep code, and public_receipts' explicit column list (verified in 0001_init.sql:168-183) means no web leak. (3) B's re-sign-after-expiry check omits searchTransactionHistory:true — getSignatureStatuses' default cache only covers ~150 recent slots, so a withdrawal confirmed minutes before a crash reads 'absent' and gets re-signed: a double-send. A specifies it; adopt verbatim. (4) /claimdeposit <sig> is a UX dead-end (users don't know tx sigs, and it only credits when the sender already matches their link — the case the watcher auto-handles); A's auto-credit of prior orphan deposits at /wallet link time solves the actual failure mode (deposited before linking). SHARED HOLES BOTH MISSED: getSignaturesForAddress returns max 1000 newest-first — anyone can dust-spam the public treasury address on devnet and push real deposits past the window, so the scanner needs a before-pagination loop until the cursor sig is found, plus a minimum-credit threshold; the watcher must also provably ignore withdrawal txs (treasury as SOURCE) via a fixture test; wallet-link squatting (link a groupmate's known pubkey first, capture their deposits) has no v1 fix without a signing page — accept, document, add admin unlink; deposit-credited notifications can't be DM'd to users who never started the bot — route to a stored last-interaction group. Verdict: B's skeleton (separate wager_* tables, Postgres RPC atomicity with pg_advisory_xact_lock, persisted wager_status breaker, outbox with pre-broadcast persisted signed bytes, module-null degrade, 7 one-line seams) corrected with A's mechanics (currency-at-mint column, (tx_sig,ix_index) deposit keying, searchTransactionHistory:true, orphan auto-credit, stream_cursors reuse, Number.isSafeInteger boundary asserts).

## Final design (synthesis)

WAGER MODE v1 — SYNTHESIS (B's parallel subsystem, A's mechanics)

PRINCIPLE: everything money-related lives in new wager_* tables and a self-contained apps/engine/src/wager/ module reached through a nullable WagerModule port (the proofSubmitter pattern). Exactly ONE change to an existing table: markets gains `currency text not null default 'rep' check (currency in ('rep','sol'))`, stamped atomically at mint. Rep code paths are byte-identical to main; deleting migration 0002 + the wager module restores main behavior.

SCHEMA (packages/db/migrations/0002_wager.sql, additive):
- alter table markets add column currency ... default 'rep' (the only ALTER).
- wager_groups(group_id PK→groups, enabled bool, enabled_by, updated_at).
- wager_wallet_links(user_id PK→users, pubkey text unique, last_wager_group_id bigint — NOTIFICATION routing only, never fund routing —, verified_at, created_at).
- wager_ledger_entries(id bigserial PK, user_id, group_id nullable, market_id nullable, kind check in ('deposit','stake','payout','refund','withdrawal','withdrawal_refund'), lamports bigint signed, idempotency_key text unique, created_at). Balance is USER-GLOBAL (sum by user_id); deposits are not group-scoped.
- wager_deposits(id bigserial PK, tx_sig text, ix_index int, sender_pubkey, lamports bigint, slot, user_id nullable = orphan, credited_at, observed_at, UNIQUE(tx_sig, ix_index)).
- wager_withdrawals outbox(id uuid PK, user_id, dest_pubkey, lamports, state check in ('debited','submitted','confirmed','failed'), tx_sig, raw_tx_b64, last_valid_block_height, error, timestamps).
- wager_settlements_applied(market_id PK→markets, applied_at) — money-movement marker, separate from settlements.posted_at (which is chat-only).
- wager_status(single row: paused bool, reason) — PERSISTED circuit breaker.
- RLS enabled on every wager_* table, ZERO anon policies, ZERO view changes, NEVER added to supabase_realtime — wallet addresses and per-user SOL amounts structurally unreachable from web.
- Two security-definer RPCs giving the atomicity supabase-js REST lacks: wager_stake(user, group, market, side, lamports, multiplier, state, placed_at_ms) — pg_advisory_xact_lock(hash('wager:user:'||user)); inside ONE transaction: balance sum ≥ stake, one-side-per-user + per-market cap over non-void positions, per-market worst-case-liability cap using mult_milli = round(multiplier*1000) (worst case = max over sides of Σ floor(stake*mult_milli/1000) − Σ stakes), wager_status.paused check, then insert positions row (shared table — reducer/Settler lifecycle unchanged) + wager_ledger_entries stake debit key 'wager:stake:'||position_id; returns {ok, position_id} or typed code insufficient|wrong_side|cap|liability_cap|paused. wager_request_withdrawal(user, lamports) — same lock; balance check; insert outbox row state 'debited' + ledger debit key 'wager:withdrawal:'||id in one transaction; dest_pubkey copied from the wallet link inside the function (never caller-supplied).

CHAIN I/O (packages/solana — pure, no DB knowledge, house never-throw result objects):
- transfer.ts: buildSolTransfer({from keypair, to, lamports, recentBlockhash, lastValidBlockHeight}) → {rawTxB64, sig} signed locally, sig known PRE-broadcast; broadcastRawTx(conn, rawTxB64) — identical bytes = identical sig = rebroadcast always safe; getSigStatus(conn, sig) using getSignatureStatuses with searchTransactionHistory:true (MANDATORY — the default cache only covers recent slots and its absence would cause double-sends after crashes); isBlockheightExceeded(conn, lvbh).
- deposits.ts: fetchIncomingTransfers(conn, treasury, {untilSig}) at 'finalized' commitment; getSignaturesForAddress with a before-pagination loop until untilSig is found (defends against >1000-sig dust-spam window eviction); batched getParsedTransactions; extract only system-program parsed {type:'transfer'} instructions where destination === treasury (withdrawals have treasury as source and are skipped — fixture-tested); emit {sig, ixIndex, sender, lamports, slot}.
- rpc.ts: withRetry — jittered backoff, 429-aware, 4 attempts, wrapped around every wager-path RPC call.

ENGINE MODULE (apps/engine/src/wager/): port.ts freezes the contract — interface WagerModule { currencyForMint(groupId): Promise<'rep'|'sol'>; handleStakeTap(args:{market: MarketRow; userId; userName; side; presetIndex; inPlay; nowMs}): Promise<{reply: string; placed: boolean}>; applySettlement(marketId): Promise<void>; settlementPayoutsLine(marketId, outcome): Promise<string>; cardFooter(): string; presetLabels(): [string,string,string]; setGroupEnabled(groupId, enabled, byUserId); isGroupEnabled(groupId); registerCommands(bot); registerCrons(registry); } — Deps gains `wager: WagerModule | null` (null when WAGER_MODE_ENABLED !== 'true' or WAGER_TREASURY_KEYPAIR_B58 absent; every seam short-circuits to exact main behavior). Dedicated plain-SOL keypair — the TxL-holding SOLANA_KEYPAIR_B58 never touches wager flows (sponsor terms: TxL never collateral). Long-lived Connection.

FLOWS:
- Deposit: /wallet <pubkey> (base58→32 bytes via packages/solana codecs; pubkey unique, first-link-wins; on link, auto-credit any prior unattributed wager_deposits rows from that sender with keys 'wager:deposit:<sig>:<ix>' — resolves deposited-before-linking; warn that re-linking moves future attribution). /deposit replies with treasury address + devnet-only warning; user sends a plain Phantom devnet transfer, no memo, no dapp page. Watcher cron (30s, pg-advisory-try-lock singleton): cursor at stream_cursors['wager:deposits:<treasuryPubkey>'] (pubkey-scoped so treasury rotation restarts scanning); process oldest→newest; per transfer upsert wager_deposits on (tx_sig, ix_index); if sender links to a user and lamports ≥ MIN_DEPOSIT_LAMPORTS, post ledger deposit (+lamports, key 'wager:deposit:<sig>:<ix>'), set credited_at + verified_at, notify in last_wager_group_id (group post, never DM — bot cannot DM users who never started it); else store as orphan. Advance cursor per processed signature — re-processing is idempotent via both unique keys.
- Stake: handleStake runs its existing checks; immediately after loading the market row, `if (market.currency === 'sol') { delegate to wager.handleStakeTap; return; }` — one line, before any Rep balance/preset logic. Wager path: unlinked → onboarding copy; presetIndex → PRESET_STAKES_LAMPORTS [0.01, 0.05, 0.1 SOL]; lock multiplier exactly as Rep (quote_multiplier / doubtMultiplier); call wager_stake RPC; map typed errors to copy. Positions are shared rows so the reducer's 150s pending window, delay-snipe voids, and activation apply with zero changes — funds already escrowed at tap, no wallet round-trip inside timing windows. Callback encoding st:<mkt>:<side>:<idx> unchanged. Rep path keeps its existing (main-identical) behavior including its benign TOCTOU.
- Mint: createMarketFromClaim computes `currency = (await deps.wager?.currencyForMint(groupId)) ?? 'rep'` and includes it in the insertMarket input — atomic stamp, no crash window; toggling the group flag never changes live markets.
- Settle: in settle(), after insertSettlement, branch: market.currency === 'sol' → skip the Rep postLedger loops, call wager.applySettlement(marketId) and use settlementPayoutsLine for the chat receipt (SOL amounts chat-only; public_receipts untouched). applySettlement is a pure function of settlements+positions, safe to run N times: refund still-pending and everyone-on-void (+stake, key 'wager:refund:<positionId>', flip pending→void), payouts per position floor(stake × round(locked_multiplier×1000) / 1000) in BigInt summed per user (key 'wager:payout:<marketId>:<userId>'), then insert wager_settlements_applied. A wager cron sweeps settled/voided sol markets lacking the marker and re-runs applySettlement — closing the settle() crash window for SOL without touching Rep's known (main-identical) window.
- Withdraw: /withdraw <amount|all> → wager_request_withdrawal RPC (min 0.01 SOL). Executor cron (15s, singleton): 'debited' rows → sign, persist {tx_sig, raw_tx_b64, lvbh, state:'submitted'} BEFORE broadcast, then broadcast; 'submitted' rows → getSigStatus (searchTransactionHistory:true): confirmed/finalized → 'confirmed' + explorer-link chat post; unknown AND blockheight exceeded → re-sign fresh (same row, new sig/raw/lvbh); unknown, not expired → rebroadcast identical bytes (same-sig safe). Terminal failure → 'failed' + ledger credit kind 'withdrawal_refund' key 'wager:wrefund:<id>'. Recipient gets exact lamports; ~5000-lamport fee treasury-absorbed inside FEE_BUFFER.
- Solvency: per-market liability cap inside the RPC; solvency cron (5 min): invariant treasury_lamports ≥ Σ wager_ledger_entries + Σ open sol-market worst-case liabilities + FEE_BUFFER_LAMPORTS; violated → persist wager_status.paused (blocks NEW stakes only — settlement credits and withdrawals never pause), requestAirdrop with retry toward FLOAT_TARGET (failure = warn), alert WAGER_OPS_CHAT_ID.

CONSTANTS (apps/engine/src/wager/constants.ts): PRESET_STAKES_LAMPORTS [10_000_000n, 50_000_000n, 100_000_000n], PER_MARKET_STAKE_CAP_LAMPORTS 100_000_000n, MAX_MARKET_LIABILITY_LAMPORTS 2_000_000_000n, MIN_DEPOSIT_LAMPORTS 1_000_000n, MIN_WITHDRAWAL_LAMPORTS 10_000_000n, MULT_SCALE 1000 (single source, asserted equal between RPC SQL and JS by a test), FEE_BUFFER_LAMPORTS 50_000_000n, DEPOSIT_POLL_MS 30_000, OUTBOX_TICK_MS 15_000, DEPOSIT_COMMITMENT 'finalized'. All timing tunables (PENDING_TAP_WINDOW_MS etc.) inherit — packages/market-engine untouched. All lamport arithmetic in bigint; Number.isSafeInteger asserts at every PostgREST boundary.

ENV: WAGER_MODE_ENABLED, WAGER_TREASURY_KEYPAIR_B58, WAGER_OPS_CHAT_ID (optional). COPY: every wager string lives only in wager/copy.ts (B's copy set: toggle explainers, /wallet, /deposit with devnet-only warning, insufficient, liability 'fully loaded', paused, receipt line '(devnet)', withdraw confirm/fail, unlinked onboarding); emission sites all behind market.currency/'group enabled' gates. COMPLIANCE TESTS: (a) existing deny-list suites (copy.test.ts, cards.test.ts, testkit.ts, spec-terms.test.ts) run unchanged and green; (b) flag-off full stake→settle cycle emits zero deny-listed tokens and zero wager DB writes; (c) import-boundary check: no module outside wager/ imports wager/copy.ts; (d) seed/topup isolation: ensureUserSeen + runMatchdayTopup write zero wager_ledger_entries rows (structural — different table — plus test); (e) anon-client leak test: selects on all wager_* tables return nothing, public view column lists contain no pubkey/lamport fields. SINGLE-INSTANCE hedges: watcher/executor crons take pg advisory try-locks so rolling-deploy overlap cannot double-run; stake/withdraw atomicity is in-DB, immune to multi-instance.

## Open risks (accepted for devnet v1)

- Wallet-link squatting: no ownership proof in v1 — an attacker who links a groupmate's known Phantom pubkey first captures that user's deposits (first-link-wins, unique pubkey). Accepted for devnet; mitigate with an admin/ops unlink command and explicit /wallet copy; the real fix (sign-a-nonce page verifiable with the existing @noble/curves ed25519 in packages/solana/src/verify.ts) is v2.
- Custodial hot wallet on Railway: the wager treasury keypair sits in an env var on a PaaS; a leak drains the float. Devnet-worthless today, but the design's escrow shape concentrates all custody here — must be re-evaluated before any non-devnet future.
- Devnet operational flakiness: requestAirdrop rate limits can leave the solvency breaker stuck paused with no automatic recovery (manual faucet top-up is the fallback); api.devnet.solana.com 429s under polling are mitigated but not eliminated by withRetry; getParsedTransactions batches are heavy calls on public RPC.
- Rolling-deploy double-instance windows: stake/withdraw atomicity is in-DB (immune), and watcher/executor crons take pg advisory try-locks, but chat side effects (deposit notifications, receipts) can still emit twice during an overlap — cosmetic, not monetary.
- PostgREST returns bigint as JS numbers: safe under 2^53 with the specified caps and Number.isSafeInteger asserts, but a future cap increase past ~9e15 lamports would silently corrupt — the asserts must fail loud, and the RPC jsonb returns must keep sums within bounds.
- Rep path retains main's known defects by design (unlocked handleStake TOCTOU, settle() ledger crash window swept only for chat): correct for contamination-avoidance on this branch, but anyone later promoting wager mode must not mistake Rep's tolerances for the SOL path's guarantees.
- Sponsor-terms optics: even with a dedicated plain-SOL keypair, the deploy runs both the TxL subscription wallet and a wagering treasury in one process; keep the branch strictly out of the hackathon submission and the TxL wallet code path physically separate (already enforced by module boundaries).
- supabase migration execution: security-definer RPCs and the markets ALTER must be applied to a devnet-dedicated Supabase project, never the instance backing the hackathon demo, or the shared schema drifts from main's migrations.
